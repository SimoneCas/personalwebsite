Coherence 12c, Planning a Successful Deployment

Ottobre 31, 2019

Link al White Paper di Oracle nel quale si descrivono tutte le configurazioni e i check da attuare in produzione per rilasciare soluzioni con Oracle Coherence 12c:
White Paper

Oltre agli aspetti di configurazione descritti nel White Paper, durante lo sviluppo tenere sempre in considerazione le seguenti linee guida:

  1. Minimizzare gli hop di rete tra nodi applicativi e nodi del datagrid
  2. La 'Key Affinity' assicura che i dati siano co-localizzati nella stessa partizione (e non solo nello stesso nodo)
  3. Durante l'esecuzione di un Entry Processor, il lock avviene solo sulla entry oggetto di modifica e su eventuali altre entry accedute tramite BackingMap

Elenco delle attività di Operation che richiedono il riavvio dei server:
Area Sistema Coherence Attività Tipologia di riavvio
Network Nodo del Cluster Cambio del trasport protocol tra Cluster Members (Cache Servers / Proxy Servers) Riavvio completo

Network Nodo del Cluster Cambio dell'IP di un cluster member non in WKA Rolling Restart

POF Serialization di Inner Class

Giugno 06, 2019

Se utilizzando Coherence si decide di serializzare gli oggetti tramite Portable Object Format (POF) Serialization bisognerà dichiarare all'interno del file pof-config.xml il mapping tra la classe da serializzare ed il relativo serializzatore. Se la classe da serializzare è una Inner Class dovremo utilizzare la seguente sintassi per indicare la classe oggetto di serializzazione: 

<user-type>
    <type-id>1001</type-id>
    <class-name>it.sample.model.Bet$BetType</class-name>
    <serializer>
       <class-name>com.tangosol.io.pof.EnumPofSerializer</class-name>
    </serializer>
</user-type>
Nell'esempio precedente viene utilizzato il carattere '$' per indicare che BetType è una InnerClass dichiarata all'interno della classe Bet.

Corretta definizione dei Listeners Coherence

Febbraio 19, 2019

Se utilizzando Coherence si ha la necessità di catturare eventi di modifica della cache, si dovrà registrare alla NamedCache un Listener Coherence con eventualmente un Filtro per rimanere in ascolto solo di eventi relativi ad un sottoinsieme di entry presenti in cache. Un esempio di registrazione di Listener Coherence: 

CacheFactory.getTypedCache(cacheName, TypeAssertion.withTypes(keyClass, valueClass)).addMapListener(listener, new Filter(), false);
Al fine di evitare problematiche di Listener duplicati nel Cluster Coherence ricordarsi sempre di cancellare il listener durante lo stop della propria applicazione; inoltre i listener registrati ed eventuali filtri utilizzati dovranno ri-definire i metodi equals e hashCode al fine di gestire correttamente lato Cluster la fase di registrazione e cancellazione dei listeners.

Abilitare jmx per Coherence

Febbraio 15, 2019

Per abilitare il monitoraggio degli mBean Coherence tramite la specifica jmx bisogna lanciare i Managed Server data storage con una delle due seguenti properties: 

-Dcoherence.management=local-only   //monitoraggio abilitato localmente
-Dcoherence.management=all          //monitoraggio abilitato da remoto
Verificare che non siano presenti le seguenti properties che renderebbero non visibili alcuni mBean di Coherence: 
-Dcom.sun.management.jmxremote=true 
-Dcom.sun.management.jmxremote.ssl=false 
-Dcom.sun.management.jmxremote.authenticate=false 
-Dcom.sun.management.jmxremote.port=10039

Weblogic JMS - Load Balancing

Ottobre 25, 2018

Se si sviluppa un'applicazione con Producer JMS in un cluster Weblogic formato da più istanze di Managed Server la configurazione di default della Connection Factory JMS abiliterà solo accodamenti "single node", ovvero ogni Producer scriverà sulla destinazione locale alla sua istanza, non sfruttando così il parallelismo delle esecuzioni multi server.

Per abilitare un accodamento bilanciato su tutto il cluster bisogna valorizzare le seguenti configurazioni come segue:

  • Server Affinity Enabled --> "False"
  • Load Balancing Enabled --> "True"


Esempio di script WLST: 
...
cmo=cd('/JMSSystemResources/'+JMS_MODULE_NAME+'/JMSResource/+JMS_MODULE_NAME+
    '/ConnectionFactories/'+CONN_FACTORY_NAME/LoadBalancingParams/'+CONN_FACTORY_NAME)
cmo.setServerAffinityEnabled(false)
cmo.setLoadBalancingEnabled(true)
...

Di seguito il link alla documentazione Oracle: https://docs.oracle.com

Weblogic Maven Dependency

Maggio 10, 2018

Ad oggi il repository centrale maven di Oracle non risulta stabile per le dipendenze di Weblogic.
Al fine di risolvere correttamente tutte le dipendenze maven di Oracle Weblogic la best practice da documentazione è quella di popolare il repository locale maven con le dipendenze di Weblogic a partira da un'installazione locale.

Guida
In breve, installare e configurare il weblogic-maven-plugin:

  • Posizionarsi nella cartella ORACLE_HOME\oracle_common\plugins\maven\com\oracle\maven\oracle-maven-sync\12.2.1
  • Lanciare il comando mvn install:install-file -DpomFile=oracle-maven-sync-12.2.1.pom -Dfile=oracle-maven-sync-12.2.1.jar
  • Lanciare il comando mvn com.oracle.maven:oracle-maven-sync:push -DoracleHome=c:\oracle\middleware\oracle_home\
Verificare la configurazione del repository locale: mvn help:describe -DgroupId=com.oracle.weblogic -DartifactId=weblogic-maven-plugin -Dversion=12.2.1-0-0

OSB Best Practice

Gennaio 30, 2016

Principali linee guida per lo sviluppo e la progettazione di logiche d'integrazione tramite Oracle Service Bus:

  1. Non cercare di sviluppare l'intera soluzione d'integration solo su OSB: OSB è solo una parte di Oracle SOA Suite, anche se installato separatamente. In alcuni progetti si tende a sviluppare tutte le soluzioni d'integrazione tramite OSB; accentrare in OSB operazioni come la gestione degli errori, delle operazioni asincrone e di logica di business può generare enorme caos. OSB è un prodotto ESB e come tale deve essere utilizzato solo per sviluppare componenti ESB.
  2. Non progettare transazioni statefull su OSB. OSB è un prodotto per la gestione di sessioni stateless, demandando a BPEL o BPM queste funzionalità.
  3. Non utilizzare le action di OSB per effettuare operazioni di INSERT/UPDATE/DELETE o chiamate a procedure. Limitarsi ad operazionni di SELECT.
  4. Non utilizzare Java Callout per implementare complessa logica di business
  5. Tra l'uso di una Route o di una Service Callout, prediligere sempre l'utilizzo della route che è implementata con una gestione dei Thread più performante, utilizzando un minor numero di risorse.
  6. Non sviluppare molteplici operazioni in un singolo stage del Proxy Service, poiché in caso di errore o di debug lo stage è il blocco più atomico identificato dal prodotto.
  7. Non processare file troppo grandi con OSB.
  8. Nell'uso dell' XPath non utilizzare wild card come * e //
  9. Impostare il progetto con un'alberatura di cartelle che permette di sudddividere il deploy in più fasi per permettere installazioni progressive ed agili.

Per ulteriori informazioni sul prodotto si rimanda alla pagina oracle service bus.
