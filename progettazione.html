Code Review

Gennaio 18, 2022

Checklist Code Review per applicazioni distribuite:

  1. Verificare che nell'integrazion con sistemi esterni siano definiti flussi che gestiscono gli errori
  2. Verificare che sia presente un piano di recovery a seguito di fail o shutdown (restore di flussi che erano attivi)
  3. Verificare che siano sempre configurati timeout nelle chiamate ai sistemi esterni e retry in caso di timeout
  4. Verificare che si utilizzi il Circuit Breaker
  5. Verificare che il numero di interazioni con sistemi esterni sia "lecito" con chiamate singole e verificare se convertirle in chiamate "massive" (aggregate)
  6. Verificare che le API siano idempotenti
  7. Verificare l'utilizzo di Cache dove possibile
  8. Vericare la configurabilità dell'applicazione: quali da db, quali da file interno, quali da file esterno, quali da System Property
  9. Verificare correttezza pom
  10. Verificare code quality e vulnerabilità tramite SonarQube

Sistemi distribuiti

Novembre 05, 2018

In fase di progettazione della base di dati, a seconda dei diversi requisiti, si dovrà scegliere quale "famiglia" di prodotto utilizzare.
Prima di elencare le principali soluzioni software ricapitoliamo le proprietà di un sistema distribuito:

Consistenza: tutti i nodi nella rete vedono lo stesso valore del dato nello stesso istante; assicurata quindi l'integrità del dato trai i diversi nodi del sistema distribuito.

Disponibilità: il servizio risulta sempre disponibile garantendo risposte ai client.

Tolleranza della partizione: il servizio continua ad operare correttamente anche in presenza di perdita di messaggi o di partizioni di rete.

Un sistema distribuito può soddisfare solo due delle tre proprietà descritte.



Domande principali a cui rispondere per poter scegliere la miglior base dati:

  • Quante relazioni esistono tra i dati?
  • Quanto è complesso il dato salvato?
  • Quanto spesso i dati cambiano?
  • Quanto spesso i dati vengono interrogati? Durante le interrogazioni si considerano le relazioni tra dati?
  • Quanto è critica l'applicazione in uno scenario di disastro?
Caratteristiche di un DB Relazionale:
  • Ottimizzato per le scritture, offre consistency e availability
  • Vantaggi: semplicità di utilizzo, facilità di query, data-integrity
  • Svantaggi: limiti nella grandezza dei campi, difficili da far scalare
Caratteristiche di un DB non Relazionale:
  • Ottimizzato per le letture, offrono availability e partition tolerance oppure consistency e partition tolerance
  • Vantaggi: flessibilità nella gestione dei dati, scalabilità
  • Svantaggi: data consistency, mancata standardizzazione
Tipologie di DB non Relazionali: Caratteristiche di un DB Relazionale:
  • Key-Value: funziona meglio con uno schema di database semplice. È meglio per molte letture, scritture e pochi aggiornamenti
  • Document: funziona meglio se hai bisogno di uno schema flessibile. I dati vengono archiviati in formato XML o JSON. Offre prestazioni di lettura elevate e puoi bilanciare le prestazioni di lettura con le prestazioni di scrittura
  • Graph: ottimo quando si dispone di uno schema di database complesso da navigare tra le relazioni degli oggetti

Marzo 14, 2017

In progetti d'integrazione aspetti trasversali al business come la gestione degli errori rivestono un ruolo fondamentale.
La presentazione descrive le principali pratiche da adottare in contesti di integrazione ESB, SOA o BPM per poter prevenire e gestire gli errori. In particolar modo saranno trattati 4 punti principali:

  • Integrazione con servizi a scalabilità limitata
  • Integrazione con servizi non raggiungibili
