Ottimizzazioni Applicationi con RDBMS

Aprile 26, 2018

Assicurare elevate perfomance con JDBC in Weblogic

  • Utilizzare Datasource Weblogic: generare connesioni JDBC è costoso, utilizzando i datasource Weblogic assicura un riuso delle connessioni; la miglior modalità di utilizzo è quella di utilizzare le connessioni per breve tempo, resituendo così velocemente al pool la connessione utilizzata al fine di aumentare il parallelismo delle interazioni.
  • Preferire l'utilizzo dell'Oracle JDBC thin driver (Type 4) al JDBC OCI driver. WebLogic ufficialmente non supporta più l'utilizzo del driver in modalità OCI.
  • Utilizzare la classe PreparedStatements invece di Statements.
  • Chiudere tutte le risorse JDBC il prima possibile: utilizzando Connection.close() saranno chiusi anche tutti i sotto-oggetti JDBC.
  • Configurare la frequenza di Datasource Shrink a zero: il datasource in assenza di richieste tende a chiudere le connesioni fino ad arrivare al numero minimo di connessioni pronte nel pool; impostando a zero la frequenza di Shrink il datasource manterrà sempre attive le connessioni create per garantire disponibilità più veloce delle connessioni.
  • Configurare il massimo di numero di Thread eseguibili dal WorkManager con il numero di connessioni massime lavorabili dal database.

Esempio di WebLogic JDBC best practices: 

Public void myTopLevelJDBCMethod() {
    Connection c = null; // defined as a method-level object, not accessible or kept where other threads can use it.
      
    … do all pre-JDBC stuff…
 
    // The try block, in which all JDBC for this method (and sub-methods) will be done
    Try {
      // Get the connection directly, fresh from a WLS datasource
      c = myDatasource.getConnection();
 
      … do all your JDBC… You can pass the connection to sub-methodshod
      doMyJDBCSubTaskWith( c );
      
      c.close(); // close the connection as soon as all JDBC is done
      c = null;  // so the finally block knows it’s been closed if it was ever obtained.
 
      .. do whatever else that may remain that doesn’t need JDBC. 
      ..I have seen *huge* concurrency improvements by closing the connection ASAP
    
    } catch (Exception e) {
      .. do what you want/need, if you need a catch-block, but *always* have the finally block:
    } finally {
      // If we got here somehow without closing c, do it now, without fail, as the first thing in the finally block so it always happens
      If (c != null) try {c.close();} catch (Exception ignore){}
      … do whatever else you want in the finally block
    }
}

Ottimizzazioni generiche: Al fine di sviluppare applicazioni performanti che utilizzano RDBMS, condivido le principali best practices da adottare in termini di sviluppo e configurazione.

  • Velocizzare la connettività verso il Database: stabilire la connessione verso il database è l'operazione più costosa. Possibili soluzioni:
    • Connection Pools lato Client
    • Proxy Connection Pools
    • Posticipare il controllo dello stato della connessione
    • De-prioritizzazione dei nodi guasti

  • Ottimizzazione dell'elaborazione delle istruzioni: La COMMIT policy predefinita con JDBC è Auto-COMMIT; a meno che ciò corrisponda al volere dello sviluppatore, si dovrebbe disabilitare esplicitamente Auto-COMMIT sulla connessione.
    • Statements Caching
    • ResultSets Caching with Change Notification - the Hard Way (JDBC-OCI)
    • ResultSets Caching with Change Notification - the Easy Way (JDBC Thin or OCI with DB 18c)

  • Ottimizzazione del traffico di rete
    • Network Data Compression
    • Sessions Multiplexing

Sql Join

Ottobre 13, 2015

In una base di dati relazionale si possono realizzare diverse tipologie di relazioni tra le tabelle, andando a definire quindi diverse categorie di join.
In particolare, il JOIN è una clausola del linguaggio SQL che serve a combinare (unire) le tuple di due o più relazioni di un database tramite l'operazione di congiunzione (od unione) dell'algebra relazionale. Lo standard ANSI definisce alcune specifiche per il linguaggio SQL sul tipo di JOIN da effettuare: INNER, FULL, LEFT e RIGHT. In alcuni casi è possibile che una tabella possa essere combinata con se stessa, in questo caso si parlerà di self-join.
La totale comprensione delle differenze tra esse permette di evitare inconvenienti ed errori di relazione durante la fase di sviluppo e test.
